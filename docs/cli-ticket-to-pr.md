# CLI агент: тікет → PR (Task E, bonus)

Тікет: `materials/task-feature-ticket.md`

## CLI-агент

Claude Code — запущений як агент, вбудований у VS Code через нативне
розширення (та сама модель і той самий агентний цикл, що й у «сирому»
`claude -p` в терміналі, інший лише транспорт).

## Команда, яку запустили

Еквівалент запропонованої в тікеті команди, введений у чат VS Code
розширення:

```
implement materials/task-feature-ticket.md
```

Тікет-файл був єдиним вхідним промптом — без додаткового переказування, як і
задумано в примітці «Task E, bonus».

Реалізація `capitalizeWords()` і тести до неї вже існували в робочій копії
на момент цього запуску (лишались незакомічені з попередньої сесії). У цьому
проході агент: (1) перевірив, що код і тести відповідають усім прикладам і
acceptance criteria з тікета, (2) прогнав `npm test`, (3) закомітив **лише**
два файли тікета (`app/src/text-utils.ts`,
`app/src/text-utils.test.ts`) — без сторонніх незакомічених правок в
`app/AGENTS.md` (Task C) і чернеток `docs/*.md` (Task A/B), які на той
момент лежали в тій самій робочій копії, (4) запушив гілку
`ws03/lsamoilenko` у форк і відкрив PR через `gh pr create`.

## Що реалізовано

Додано `capitalizeWords(input: string): string` у `app/src/text-utils.ts`
поруч із `slugify`/`truncate`/`parseTags`: перша літера кожного слова —
велика, решта — маленькі; пробіли (включно з множинними, початковими й
кінцевими) копіюються з вхідного рядка без змін через `\S+`-заміну замість
`split`/`join` по пробілу. Додано 6 unit-тестів у
`app/src/text-utils.test.ts`, що покривають усі приклади з тікета
(`"WIRELESS mouse"`, `"red   t-shirt"`, обрізку пробілів, порожній рядок,
рядок із самих пробілів, уже-Title-Case вхід). Наявні тести
`slugify`/`truncate`/`parseTags` не змінювались.

## Спостереження: CLI vs IDE-агент (Task B)

У Task B (`comparison.md`) Claude Code запускався як термінальний CLI-агент
проти Cursor як IDE-агента; тут та сама модель Claude Code відпрацювала
всередині IDE-чату — різниця в UX виявилась мінімальною: той самий
пошук/читання файлів, точкові `Edit`-правки, самостійний запуск `npm test`
для верифікації. Головна різниця не «CLI vs IDE-агент», а **де саме
проходить межа підтвердження для git-дій**: у «сирому» `claude -p` в
терміналі агент міг би одним проходом створити гілку, закомітити й
відкрити PR без паузи; тут, у чаті, редагування коду й git-операції
(коміт, пуш, `gh pr create`) — явно санкціоновані окремі кроки, що дало
змогу відсіяти сторонні незакомічені зміни (Task C/A/B чернетки) з diff'а
цього PR і закомітити рівно те, що стосується тікета. Для CI/«тікет → PR»
без людини в циклі термінальний `claude -p` лишається зручнішим і швидшим;
IDE-чат кращий, коли робоча копія «брудна» (кілька задач одночасно) і
потрібен контроль над тим, що саме потрапляє в коміт/PR.
