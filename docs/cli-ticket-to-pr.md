# CLI-агент: тікет → PR (Task E, bonus)

Тікет: `materials/task-feature-ticket.md` (FEAT-102 — додати `capitalizeWords()`).

## CLI-агент

**Claude Code CLI** — свіжа агент-сесія (Sonnet-клас), якій на вхід дали
**лише шлях до тікета**, без переказування задачі своїми словами. Це той самий
інструмент, що виявився найшвидшим/найдешевшим у Task B
(`docs/comparison.md` — Claude Code CLI, Sonnet 5, 34с API, $0.24).

## Команда, яку запустили

Патерн `тікет → код → PR` — агенту передається один файл як єдиний вхід:

```bash
claude -p "implement the ticket in materials/task-feature-ticket.md"
```

У цьому прогоні я запустив це як **дві паралельні свіжі агент-сесії Claude
Code з ідентичним промптом** — щоб поставити контрольований експеримент навколо
`app/AGENTS.md` (див. розділ «Бонус» нижче). Кожна сесія сама прочитала тікет,
написала код і тести й запустила `npm test`. Git-операції (гілка, коміт, PR)
я зробив як оркестратор — агентам git було заборонено, щоб diff лишався чистим.

## Що реалізовано

Додано `capitalizeWords(input: string): string` у `app/src/text-utils.ts` поруч
із `slugify`/`truncate`/`parseTags`/`wordCount`. Реалізація — чистий однорядковий
підхід, що переписує лише послідовності не-пробільних символів, тож **пробіли
(кількість, позиції, ведучі/кінцеві, таби/переноси) зберігаються байт-у-байт**,
змінюється тільки регістр літер:

```ts
export function capitalizeWords(input: string): string {
  return input.replace(/\S+/g, (word) => word.charAt(0).toUpperCase() + word.slice(1).toLowerCase());
}
```

Тести: усі 5 рядків із таблиці тікета + edge cases (порожній рядок, рядок із
самих пробілів, змішані таби/переноси) + окремий блок характеристичних тестів
(дефіс/апостроф як одне слово, юнікод/діакритика, цифри, односимвольні слова).
`app/src/text-utils.test.ts`: **11 → 17 тестів** для `capitalizeWords`.

## Спостереження: CLI vs IDE-агент (Task B)

- **Придатність до CI / автоматизації.** Патерн «один файл-тікет → агент →
  diff» тривіально скриптується (`claude -p "…"` у пайплайні), чого IDE-агент
  із Task B (Antigravity/Copilot у GUI) не дає — там кожен крок за кермом
  людини в редакторі. CLI віддає результат як звичайний working-tree diff,
  готовий до `git`/`gh`.
- **Контроль над кроками.** IDE-агент показує зміни інлайн у редакторі й дає
  приймати їх поштучно; CLI-агент повертає готовий пакет змін — рев'ю робиш
  постфактум через `git diff`. Для відтворюваного порівняння (однаковий вхід,
  паралельні прогони) CLI зручніший; для інтерактивного «веди мене за руку» —
  IDE.
- **Швидкість збігається з Task B:** обидві свіжі сесії видали робочий фікс за
  один промпт без follow-up (Agent B ≈ 1хв40с, Agent A ≈ 3хв7с wall,
  ~21–23k токенів кожна), як і Claude Code CLI на тій самій задачі-багу.

## Бонус-експеримент: чи змінює `app/AGENTS.md` поведінку агента? (A/B-контроль)

**Гіпотеза:** файл контексту `app/AGENTS.md` (артефакт Task C) має підштовхувати
агента до конвенцій проєкту. **Дизайн:** два свіжі прогони Claude Code з
**ідентичним** мінімальним промптом; єдина відмінність — наявність
`app/AGENTS.md`:

- **Agent A (realistic):** робоча гілка, `app/AGENTS.md` присутній.
- **Agent B (control):** ізольований `git worktree`, `app/AGENTS.md` **вилучено**
  (кореневий `AGENTS.md` репо лишився — тобто контроль усе одно мав homework-level
  контекст та guardrail про незмінність сигнатур, але **не** app-level конвенції
  про типи/колокацію тестів).

| Критерій | Agent A (з `app/AGENTS.md`) | Agent B (без `app/AGENTS.md`) |
|---|---|---|
| Підхід | `input.replace(/\S+/g, …)` | `input.replace(/\S+/g, …)` — **ідентичний** |
| Пробіли не схлопуються | ✅ | ✅ |
| Явні типи `(input: string): string` | ✅ | ✅ |
| Колокований тест (`*.test.ts`) | ✅ | ✅ |
| Пояснювальний коментар до інваріанта | ✅ | ✅ (**теж додав!**) |
| Кількість тест-кейсів | 6 (вкл. таби/переноси `\t`/`\n`) | 6 (пробіли без табів) |
| Перший символ слова | `word[0]` | `word.charAt(0)` |
| `npm test` | ✅ green | ✅ green |
| `npm run typecheck` (ніхто не запускав) | ❌ **fail** — `word[0]` під `noUncheckedIndexedAccess` = `string \| undefined` | ✅ **clean** — `charAt(0)` тотальний |
| Токени / tool-uses / час | ~20.6k / 8 / ~3хв7с | ~23.4k / 10 / ~1хв40с |

### Висновок експерименту

1. **Майже повна конвергенція.** Обидва прогони незалежно обрали той самий
   підхід, зберегли пробіли, поставили явні типи, колокували тести — і навіть
   **обидва** додали пояснювальний коментар. Тобто на *маленькій, внутрішньо
   узгодженій* кодовій базі з *точним* тікетом здатний агент реконструює
   конвенції з самого коду; `app/AGENTS.md` тут — страховка, а не вирішальний
   фактор. Він давав би більше, якби код був неузгоджений або тікет — розмитий.
2. **Єдина суттєва розбіжність — не там, де просили.** Обидва агенти запустили
   лише `npm test` (зелений) і на цьому зупинилися; **жоден не запустив
   `npm run typecheck`** — попри те, що `app/AGENTS.md` прямо називає його
   «найближчою перевіркою». Realistic-прогон навіть **зашипив type error**
   (`word[0]`), а control випадково його уник (`charAt(0)`) — це варіативність
   семплінгу моделі, а не заслуга `AGENTS.md`. Урок: **порада, яку не
   форсить hook/CI, ігнорується незалежно від наявності файла контексту** — це
   прямо підпирає рекомендацію Task D робити ставку на CI-enforcement, а не на
   довіру до context-файлів.

### Інженерне доопрацювання (agent drafts → engineer hardens)

- **Фікс типів:** `word[0]` → `word.charAt(0)` — тотальний під
  `noUncheckedIndexedAccess`, поведінка ідентична (матчі `\S+` завжди непорожні).
  Після фіксу `npm run typecheck` — чистий.
- **Більше кейсів:** доданий блок характеристичних тестів фіксує поведінку, яку
  тікет лишив неявною, але однозначно визначив своїм правилом (дефіс/апостроф =
  одне слово, юнікод, цифри, односимвольні слова) — щоб «розумніший» рефактор
  (split по дефісах) не змінив її нишком.
- **Mutation-check:** тимчасово підмінив реалізацію на таку, що схлопує пробіли
  (`split(/\s+/).join(" ")`) → рівно **4** тести на збереження пробілів стали
  RED (кратні пробіли, ведучі/кінцеві, самі пробіли, таби/переноси), після чого
  реалізацію відновлено. Це доводить, що набір тестів реально стереже пастку
  тікета, а не просто «зелений».
