> **\* Рядок «Enterprise-функції» відповідає на питання: чи пройде інструмент перевірку безпеки у великій компанії?
> - **SOC 2 Type II** — незалежний аудит підтвердив, що вендор реально захищає дані (не разова перевірка, а спостереження за кілька місяців). Без цього сертифіката корпоративні закупівлі зазвичай не пропускають інструмент.
> - **SSO (SAML/SCIM)** — співробітники входять через корпоративний акаунт компанії (Okta, Azure AD), а не через особисті; коли людина звільняється, доступ забирається автоматично.
> - **Data residency** — гарантія, у якій країні обробляється код (важливо для GDPR та регульованих галузей).
> - **Audit logs** — журнал «хто, коли і що робив» для розслідувань і комплаєнсу.
> - **IP indemnity** (лише Copilot) — GitHub бере на себе юридичну відповідальність, якщо згенерований код порушить чиїсь авторські права.
> - **Self-hosted** (лише JetBrains) — моделі працюють на серверах самої компанії, код взагалі не потрапляє до вендора — найсильніша відповідь на вимоги ізоляції, сильніша за будь-який сертифікат.

Рахунок 3 : 2 : 2 — розрив мінімальний, і він не означає «Claude переміг»: підсумок залежить від ваги критеріїв для конкретної команди (для enterprise-закупівлі вирішальними будуть два «копілотівські» рядки, для NDA-проєкту — приватність).

1. **Claude Code** (3 бали) — найсильніший за агентною автономністю та автоматизацією (subagents, hooks, headless-режим у CI, прозорий API-прайсинг), але єдиний із трьох без інлайн-автодоповнення.
2. **JetBrains AI + Junie** (2 бали) — виграє за глибиною IDE-інтеграції (семантичний індекс) і приватністю: єдина екосистема з локальними моделями (в AI Assistant та Junie CLI; агент Junie в IDE — лише хмарні).
3. **GitHub Copilot** (2 бали) — лідирує за enterprise-зрілістю (IP indemnity, policy-контроль), безкоштовним тіром і нативною інтеграцією з GitHub (coding agent, рев'ю в PR).

Для нашого випадку (щоденна робота в IntelliJ + агентна задача «тікет → фікс → PR») найсуттєвішою виявилась відмінність «агент у терміналі vs плагін в IDE»: жоден інструмент не закриває обидва сценарії однаково добре, тому висновок — комбінація, а не один переможець.

### Рекомендації

**Комбінуйте, а не обирайте один.** Найсильніша конфігурація — пара «агент + IDE-плагін»: CLI/агент (Claude Code) для автономних задач «тікет → фікс → PR» плюс IDE-плагін (JetBrains AI або Copilot) для інлайн-доповнення й навігації. Один інструмент цю пару не замінює: у Claude Code немає автодоповнення, а IDE-плагіни поступаються автономією в терміналі та CI.

**Для асинхронної роботи без участі розробника** всі три екосистеми мають хмарних/CI-агентів, і це варто враховувати при виборі:
- **Claude Code** — не лише CLI: web-додаток (claude.ai/code) для хмарних сесій, розширення для VS Code і JetBrains, GitHub Action (згадка `@claude` в issue/PR запускає агента в CI) та Agent SDK для власних агентів на тій самій основі.
- **GitHub Copilot** — coding agent: призначаєте GitHub issue на Copilot, агент працює у хмарному середовищі на GitHub Actions і відкриває draft PR на рев'ю (доступний з Pro+/Business).
- **JetBrains AI + Junie** — Junie CLI для CI/CD-сценаріїв з BYOK; хмарного агента формату «призначив issue — отримав PR» немає.

Розрив між «CLI-агентом» та «IDE-плагіном» швидко закривається: кожен із трьох вендорів доставляє того самого агента в термінал, IDE, хмару та CI, тож форм-фактор перестає бути головним критерієм вибору. Реальних критеріїв два.

**Критерій 1 — екосистема, в якій живе код:**

**Критерій 2 — виміряна якість агента на власних задачах:** прогнати той самий тікет через кожного кандидата й порівняти час до робочого фіксу, кількість ітерацій та чи знайдена першопричина, а не симптом. Матриці порівняння відповідають лише на питання «що інструмент вміє»; на питання «наскільки добре» відповідає тільки такий прогін.
