# Домашнє завдання — Воркшоп 3

**Тема:** Огляд та порівняння AI-інструментів розробки
**Формат:** artifacts-first — ви тестуєте кілька інструментів на одній задачі й документуєте порівняння
**Час:** ~2–2.5 години
**Здача:** Pull Request у starter-repo (CodeRabbit зробить авто-рев'ю)

---

## 0. Налаштування

```bash
# форк + клон
gh repo fork koldovsky/2026-udc-03-tools-comparison-hw --clone
cd 2026-udc-03-tools-comparison-hw

# робоча гілка
git checkout -b ws03/<github-username>

# крихітний sample-проєкт (ціль для Task B/C/E)
cd app && npm install && npm test && cd ..
```

Що потрібно: **2–3 AI-інструменти** (IDE-плагін, agentic IDE або CLI-агент —
напр. Cursor + Claude Code, або Copilot + JetBrains AI, або будь-яка інша пара/трійка
з вашого арсеналу), Node 22+, GitHub account.

> Якщо у вас є платна підписка лише на один інструмент — для другого/третього
> цілком підійде **free tier або trial**. Мета — реальне порівняння, а не
> ще одна платна підписка.

> Усі артефакти кладіть у погоджені шляхи (`docs/...`), щоб авто-рев'ю їх знайшло.

### Що засіяно в репо

| Шлях | Що це |
|---|---|
| `app/src/text-utils.ts` | Крихітний TS-модуль (`slugify`, `truncate`, `parseTags`) з підкладеним багом у `truncate()` |
| `app/src/text-utils.test.ts` | Наявні тести — усі зелені; баг навмисно не пійманий ними |
| `app/AGENTS.md` | Навмисно тонкий, «заточений під один інструмент» baseline (ціль Task C — узагальнити) |
| `materials/task-bug-fix.md` | Тікет з описом бага в `truncate()` — **та сама задача** для всіх інструментів у Task B |
| `materials/task-feature-ticket.md` | Невеликий Jira-подібний тікет («додати `capitalizeWords()`») — для bonus Task E |
| `docs/templates/` | Скелети для ваших звітів (`tool-matrix.md`, `comparison.md`, `agents-md-notes.md`, `recommendation.md`, `cli-ticket-to-pr.md`) |

## Task A — Матриця можливостей інструментів _(~30 хв)_

**Мета:** систематизувати, чим інструменти відрізняються один від одного — не «на око», а за конкретними критеріями.

1. Оберіть **2–3 інструменти**, які порівнюватимете (можна змішувати IDE-плагіни,
   agentic IDE та CLI-агенти).
2. Скопіюйте `docs/templates/tool-matrix.md` у `docs/tool-matrix.md`.
3. Заповніть таблицю за критеріями: вибір/якість моделі, агентна автономність та
   multi-step execution, індексація кодової бази, підтримка rules/commands/skills/MCP,
   ціна та ліміти, enterprise-функції (SSO/SOC2/data residency), приватність/політика
   навчання на даних.
4. Для кожного критерію — **конкретне** значення (модель/ліміт/факт), а не загальні
   фрази на кшталт «добре» чи «погано».
5. Напишіть короткий висновок: де відмінності виявились найсуттєвішими саме для
   вашого випадку.

**Перевірка:** `docs/tool-matrix.md` існує, заповнений по всіх рядках для всіх
обраних інструментів, є висновок.

## Task B — Та сама задача в кількох інструментах _(~40 хв)_

**Мета:** порівняти інструменти на одній відтворюваній задачі, а не за відчуттями.

1. Відкрийте `materials/task-bug-fix.md` — це опис бага в `truncate()`
   (`app/src/text-utils.ts`). Перший запит до інструмента — **вставте тікет як
   є**, без переказування своїми словами (це і є задача, яку порівнюєте).
2. Запустіть `cd app && npm test` — переконайтесь, що наявні тести **зелені**
   (баг навмисно не пійманий поточними тестами — саме тому QA знайшло його
   вручну; ваш фікс має додати регресійний тест, що ловить це надалі).
3. Виконайте задачу окремо в **кожному** з 2–3 обраних інструментів. Перед
   кожним новим інструментом обов'язково скиньте зміни до чистого стану:
   ```bash
   git checkout -- app/src/text-utils.ts
   ```
   Без цього кроку наступний інструмент побачить уже виправлений код, і
   порівняння перестане бути «яблука до яблук».
4. Для кожного інструменту зафіксуйте: час до робочого фіксу, кількість
   ітерацій/промптів, чи знайшов справжню причину бага (а не симптом), чи
   з'явились регресії, токени/вартість (якщо інструмент їх показує), UX-нотатки.
5. Скопіюйте `docs/templates/comparison.md` у `docs/comparison.md` і заповніть
   таблицю + висновок.
6. Залиште фінальний робочий фікс (від найкращого інструменту) у
   `app/src/text-utils.ts` — `npm test` має бути зеленим.

**Перевірка:** `docs/comparison.md` заповнений для всіх обраних інструментів на
**одній і тій самій** задачі; `cd app && npm test` зелений.

> Цю саму вправу можна повторити на вашому робочому проєкті (дух курсу —
> «реальна робота») — але **звіт порівняння обов'язково має лежати в
> `docs/comparison.md` цього репо**, інакше CodeRabbit його не побачить.

## Task C — Крос-tool AGENTS.md _(~25 хв)_

**Мета:** перевірити, чи можна один файл контексту зробити «спільною мовою» для різних інструментів.

1. Відкрийте `app/AGENTS.md` — він навмисно тонкий і написаний під один
   конкретний інструмент (специфічні згадки, формулювання).
2. Узагальніть його: приберіть прив'язку до одного інструменту, додайте/уточніть
   стек, команди (`build`/`test`/`lint`), 3–5 конвенцій коду, guardrails.
3. Перевірте підхоплення в **мінімум 2 інструментах**: дайте однаковий короткий
   промпт верифікації (напр. «опиши конвенції цього проєкту» або «додай новий
   util за зразком `text-utils.ts`») і подивіться, чи інструмент дотримався
   `AGENTS.md` без додаткових нагадувань.
4. Скопіюйте `docs/templates/agents-md-notes.md` у `docs/agents-md-notes.md` і
   заповніть: що змінили, таблицю підхоплення по інструментах, промпт
   верифікації, результат, висновок.

**Перевірка:** `app/AGENTS.md` оновлений і узагальнений; `docs/agents-md-notes.md`
показує перевірку у **2+** інструментах.

## Task D — Командна рекомендація _(~20 хв)_

**Мета:** синтезувати Task A+B+C у рішення, яке можна принести команді.

1. Скопіюйте `docs/templates/recommendation.md` у `docs/recommendation.md`.
2. На основі матриці (A) та результатів на задачі (B) визначте **primary**
   інструмент команди — з обґрунтуванням.
3. Визначте **secondary** інструмент і сценарії, де він кращий за primary
   (напр. великі рефактори, CLI для CI, швидкі правки без IDE).
4. Випишіть ключові trade-offs (ціна vs автономність, якість індексації vs
   приватність/enterprise-tier, портативність AGENTS.md vs глибина tool-specific
   фіч тощо).
5. Зафіксуйте один конкретний ризик/застереження, за яким варто стежити
   (vendor lock-in, зміна цінової політики, відсутність audit-логів тощо).

**Перевірка:** `docs/recommendation.md` містить обґрунтований primary/secondary
вибір, що спирається на дані з Task A та B.

## Task E (bonus) — CLI-агент: тікет → PR _(~20 хв)_

**Мета:** відчути патерн «тікет → код → PR» з агентним CLI-інструментом.

1. Оберіть CLI-агента (Claude Code, Codex CLI, Aider, Goose тощо).
2. Дайте йому `materials/task-feature-ticket.md` як вхідну задачу — нехай
   реалізує `capitalizeWords()` у `app/src/text-utils.ts` (+ тести).
3. Перевірте `cd app && npm test` — зелений.
4. Відкрийте окремий PR із цією зміною (можна в той самий форк, окрема гілка
   або коміт у межах `ws03/<username>` — головне, щоб diff було видно).
5. Скопіюйте `docs/templates/cli-ticket-to-pr.md` у `docs/cli-ticket-to-pr.md`:
   яка команда запущена, що реалізовано, посилання на PR/diff-summary,
   спостереження CLI vs IDE-агент (Task B).

**Перевірка:** `docs/cli-ticket-to-pr.md` заповнений, є посилання на PR/diff,
`capitalizeWords()` реалізована й покрита тестом.
