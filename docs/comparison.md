## Сильні та слабкі сторони

**GitHub Copilot**
- Сильні сторони:
  - Швидкі підказки й редагування безпосередньо в редакторі, підтримка змін у кількох файлах.
  - Добре генерує мінімальні робочі фікси й тести за явного запиту.
  - Органічно інтегрується у звичний робочий цикл (редагування + локальний запуск тестів).
- Слабкі сторони / зауваження:
  - Може вимагати одного-двох уточнень, щоб коректно врахувати всі приймальні критерії.
  - Оцінка токенів/вартості у цьому документі приблизна й залежить від конфігурації моделі; краще орієнтуватися на час і UX, а не на точні числа.

**Claude Code**
- Сильні сторони:
  - Знайшов справжню першопричину бага (а не симптом) без додаткових підказок і без зміни публічної сигнатури.
  - Весь цикл «read → edit → run tests» відбувається в одному потоці, без перемикання в термінал вручну.
  - Одразу додав релевантний edge-case тест (`suffix.length >= maxLength`), а не лише приклад із тікета.
  - Зелено з першого запуску `npm test` (з `app/`) — жодних повторних ітерацій на сам фікс.
- Слабкі сторони / зауваження:
  - Перше повідомлення користувача не містило прямого запиту, тож асистент витратив один промпт на уточнювальне питання, перш ніж редагувати код — за наявності явнішого початкового запиту цей крок був би зайвим.
  - Оцінка токенів приблизна й залежить від того, скільки файлів/контексту потрапило в сесію; орієнтуватись варто на час і UX, а не на точні числа.

**JetBrains AI Assistant + Junie (IDE)**
- Сильні сторони:
  - Глибока IDE‑інтеграція: план → правки → запуск `npm run typecheck`/`npm test` прямо в IDE з логами.
  - Сильний семантичний індекс: упевнено тримає крос‑файловий контекст і дотримується `app/AGENTS.md` без нагадувань.
  - Дотримується guardrails: не додає залежностей/лінтерів, не змінює публічні сигнатури, робить мінімальні дифи.
  - Зручно для «малих, перевірених змін»: швидкий фідбек у редакторі, стабільні перевірки перед завершенням.
- Слабкі сторони / зауваження:
  - Орієнтований на IDE: потребує відкритого проєкту в JetBrains; менш зручний для CLI‑сесій поза IDE.
  - Немає прозорої оцінки токенів/вартості — оцінюємо за часом і UX.
  - Іноді просить формалізувати план/критерії перед застосуванням правок (додатковий крок, але підвищує передбачуваність).

## Висновок

Усі три інструменти на цьому тікеті знайшли справжню причину бага (suffix поза
межею `maxLength`, а не симптом) і додали регресійні тести без повторних
ітерацій на сам фікс — жодних регресій. JetBrains AI Assistant + Junie (IDE)
показав стабільний IDE-цикл «план → правки → `typecheck`/`test`» із сильним
крос‑файловим контекстом і дотриманням guardrails; Claude Code витратив один
додатковий промпт на уточнення наміру; Copilot потребував короткого уточнення
приймальних критеріїв.

Практично: Copilot — для швидких локальних правок за наявності чітких критеріїв;
Claude Code — для найповнішого автономного циклу з додатковими edge‑case тестами;
JetBrains AI Assistant + Junie — зручний у IDE для малих, перевірених змін із
передбачуваними перевірками (`npm run typecheck`/`npm test`) прямо з редактора.

**Який фікс залишено в репозиторії:** фінальний код у `app/src/text-utils.ts` +
`app/src/text-utils.test.ts` — це фікс від **GitHub Copilot** (9/9 тестів,
включно з обома обов'язковими за тікетом тестами й одним додатковим). Фікси
Claude Code та Junie (по 8/8 — базові 6 тестів + 2 обов'язкових за тікетом)
перевірялись окремо на скинутому коді й у фінальний коміт не увійшли.

**Перевірка:** `cd app && npm install && npm test` (залежності не закомічені —
`npm test` з кореня репо чи без попереднього `npm install` впаде з `vitest: not
found`). Останній прогін: 15/15 зелено.
