## Скидання стану між спробами

Перед кожним новим інструментом `walkthrough.md` наказує:

```bash
git checkout -- app/src/text-utils.ts
```

Це единий явний reset-крок, і команда точна та однозначна — не залишає
простору для здогадок. Я підтверджую, що вона працює як описано: команда
відкочує лише `text-utils.ts` до версії з HEAD, не займаючи
`text-utils.test.ts` (де накопичуються регресійні тести з кожної спроби —
і це, судячи з усього, навмисно, бо тікет вимагає "додай регресійний тест"
один раз, а не по колу для кожного інструменту). **Неявне припущення**, яке
варто було б проговорити явно в `walkthrough.md`: тест-файл (`*.test.ts`) не
скидається цією командою, тож якщо другий інструмент напише тест з іншою
назвою/структурою, у файлі накопичаться тести від усіх спроб одночасно —
це нормально й навіть корисно (видно різні підходи), але неочевидно з тексту
інструкції, чи це задумана поведінка, чи учасник має вручну прибирати тести
попередніх інструментів теж. Я залишила тест-файл як є (не скидала його),
бо в моєму випадку був лише один реальний прогін.

## Реальний прогін (Claude Code, CLI-агент)

Я вставила `materials/task-bug-fix.md` як є (без переказу) як єдиний вхід.

| Інструмент | Час до робочого фіксу | Ітерацій/промптів | Якість (знайшов справжній баг? регресії?) | Токени/вартість (якщо видно) | UX-нотатки |
|---|---|---|---|---|---|
| **Claude Code (реальний прогін)** | ~2 хв (один прохід: read → edit → run tests) | 1 промпт / 1 редагування файлу (`truncate()`) + 1 редагування тестів | Так — знайшов справжню причину (suffix не враховано в межі maxLength, а не просто "результат задовгий"); фікс відповідає acceptance criteria тікета один-в-один (`"a long sentence here"`, `maxLength=10` → `"a long ..."`, довжина 10); регресій немає — `npm test` зелений (8/8, включно з 2 новими regression-тестами) | Не вимірювалось окремо в цьому прогоні (немає вбудованого token-лічильника в середовищі виконання) | Тікет був достатньо конкретний (містив точний очікуваний результат і edge case з `suffix.length >= maxLength`), тому фікс — механічна дія, а не дослідження; єдина ручна дія понад мінімум — довелось усвідомлено послабити/переписати вже наявний тест `"truncates long input and appends the suffix"`, бо він неявно кодував **баговану** поведінку (`result.startsWith("hello")` при `maxLength=5` несумісне з коректним резервуванням місця під suffix) |
| Cursor (реконструйована оцінка, НЕ живий прогін) | ~1–3 хв (орієнтовно) | Ймовірно 1 промпт — Composer/Agent mode зазвичай одразу читає файл з тікетом і застосовує diff | З високою ймовірністю так, за умови вставки тікета як є (тікет явно вказує причину бага) | Insights-панель Cursor показує вартість запиту в "fast requests", якщо увімкнено | Основна відмінність від Claude Code — inline diff-перегляд у самому редакторі перед застосуванням, що прискорює рев'ю фіксу |
| GitHub Copilot (реконструйована оцінка, НЕ живий прогін) | ~2–4 хв (орієнтовно) | Ймовірно 1–2 промпти — agent mode іноді потребує додаткового "run tests and confirm" промпту | Ймовірно так, з тим самим застереженням про якість тікета | Premium-request лічильник у статус-барі VS Code | Потребує явного дозволу на запуск терміналу (test command) в частини конфігурацій за замовчуванням, що додає один клік порівняно з повністю автономним CLI-агентом |

## Висновок

**Найважливіше застереження:** лише рядок Claude Code вище — це фактичні дані;
решта — обґрунтована реконструкція, а не вимірювання. З огляду на це, головний
валідний висновок з цього dry-run — сам тікет (`materials/task-bug-fix.md`)
написаний достатньо конкретно (точний очікуваний рядок, точна довжина, явний
edge case), що будь-який сучасний агентний інструмент, найімовірніше, впорається
з першого промпту, і різниця між інструментами в цій конкретній задачі буде
більше про UX/швидкість рев'ю, ніж про здатність знайти баг. Реальний
учасник курсу, повторюючи цю вправу, отримає змістовніше порівняння, якщо
хоча б один з обраних інструментів — слабший/дешевший tier, де відмінності в
"чи знайшов причину, а не симптом" стануть помітнішими.
