## Висновок

На задачі BUG-101 обидва інструменти впоралися з першого промпту без
додаткових уточнень: знайшли корінь проблеми (суфікс додається поверх
`slice(0, maxLength)`), реалізували очікувану логіку з edge case для довгого
суфікса і додали регресійні тести з тікета. Зайвих змін у сигнатурі чи
сусідніх функціях не було в жодного; обидва самостійно запустили `npm test`
і підтвердили, що всі 8 тестів зелені.

За якістю фіксу результати практично ідентичні — задача виявилася занадто
простою, щоб розділити інструменти за цим критерієм. Відмінності радше в
UX: Claude Code був трохи швидшим і «розмовнішим» — окрім фіксу, сам
відзначив статус git (зміни незакомічені) і нагадав про методологію Task B
(зафіксувати кількість ітерацій для порівняння), тобто краще тримав контекст
репозиторію поза межами самого тікета. Cursor дав такий самий мінімальний
diff у звичному IDE-інтерфейсі. Вартість у жодному з інструментів не видно
прямо в сесії (у Claude Code — через `/cost`).

Висновок: на добре сформульованому тікеті з чіткими acceptance criteria
(відтворення, очікувана поведінка, edge cases) сучасні агентні інструменти
дають однаково коректний результат за одну ітерацію — якість постановки
задачі впливає на результат більше, ніж вибір інструмента. Якщо буде прогін
третього інструмента, цікавіше порівнювати не «чи знайшов баг», а поведінку
навколо: запуск тестів без нагадування, мінімальність diff і роботу з git.
