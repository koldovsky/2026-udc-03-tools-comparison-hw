# Рекомендація для команди (Task D)

> **Примітка щодо методології:** ця рекомендація синтезує Task A (матриця,
> побудована на основі задокументованих характеристик інструментів) і Task B
> (де реальним, фактично виконаним прогоном був лише Claude Code — див.
> застереження на початку `docs/comparison.md`). Тому рекомендація нижче
> варта як **приклад структури висновку**, а не як остаточний вердикт,
> заснований на трьох рівноцінних емпіричних прогонах.

## Рекомендований основний (primary) інструмент

**Claude Code**

Чому: у Task B це був єдиний інструмент, який ми реально прогнали на тікеті
BUG-101 — фікс зійшовся з acceptance criteria тікета з першого промпту, без
регресій, з доданими регресійними тестами за один прохід. За матрицею
Task A він також дає найкращу комбінацію для команди, яка хоче
автоматизацію поза IDE: headless-режим (`-p`) для CI/скриптів (безпосередньо
використаний у Task E цього ж завдання), підтримка MCP та Skills для
розширення, і немає прив'язки до конкретного редактора — розробники можуть
лишатися у своєму звичному IDE й викликати агента з терміналу.

## Рекомендований додатковий (secondary) інструмент

**Cursor** — для сценаріїв: великі крос-файлові рефактори, де вбудована
індексація кодової бази (codebase embeddings) дає швидший semantic search по
великому репо, і для розробників, які хочуть бачити inline-diff у редакторі
перед застосуванням змін, а не читати diff у терміналі.

Чому: за матрицею Task A вбудований індекс Cursor — найсильніша його сторона
порівняно з Claude Code (який покладається на grep/glob у реальному часі) і
Copilot (де якість embeddings-пошуку залежить від розміру репо). Для команд,
що працюють з великою кодовою базою, а не з невеликими ізольованими
модулями (як `text-utils.ts` у цьому репо), різниця в швидкості навігації
може бути суттєвішою, ніж у цьому конкретному dry-run.

## Ключові компроміси (trade-offs)

- Ціна vs автономність: Claude Code Max ($100–200/міс) дорожчий за Cursor Pro
  ($20/міс) на одного розробника, але дає ширші ліміти для повністю
  автономних, довгих агентних сесій (наприклад, ганяти CI-скрипти) — плата
  за автономність, а не лише за доступ до моделі.
- Якість індексації vs приватність/enterprise-tier: вбудований codebase-index
  Cursor зручний, але означає, що код (або його embeddings) проходить через
  інфраструктуру Cursor за замовчуванням, якщо явно не увімкнено privacy
  mode; Claude Code без власного індексу читає файли безпосередньо з
  локальної файлової системи в кожній сесії.
- Портативність AGENTS.md vs глибина tool-specific фіч: спільний
  `app/AGENTS.md` (Task C) дає базову крос-tool узгодженість команд і
  конвенцій, але не замінює tool-specific можливості (напр. Cursor
  `.cursor/rules/*.mdc` з glob-scoped правилами або Claude Code custom
  slash-commands) — якщо потрібна тонша поведінка на рівні окремого
  інструменту, AGENTS.md варто доповнювати, а не заміняти tool-specific
  файлами.
- Enterprise-готовність vs швидкість впровадження: GitHub Copilot вигідний
  там, де компанія вже має контракт з GitHub Enterprise (SSO/SOC2 вже
  узгоджені), навіть якщо "чистий" агентний UX слабший — organizational
  overhead часто важливіший за маржинальну різницю в якості фіксу.

## Ризик/застереження, за яким варто стежити

**Vendor/model lock-in через AGENTS.md-конвенції, побудовані під одну
модель:** якщо команда напише `AGENTS.md`/промпти, налаштовані під
специфічні сильні/слабкі сторони однієї моделі (напр. Claude Sonnet), а
пізніше перейде на інший інструмент з іншою базовою моделлю, частина
неявних припущень (стиль коментарів, очікувана деталізація тестів) може не
перенестись автоматично — тому варто раз на квартал повторювати "verification
prompt" з Task C на реальних 2+ інструментах, а не покладатися на
одноразову перевірку при написанні AGENTS.md.
