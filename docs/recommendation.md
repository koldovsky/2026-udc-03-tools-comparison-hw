# Рекомендація для команди (Task D)

## Рекомендований основний (primary) інструмент

**Claude Code**

Чому: за матрицею з Task A Claude Code має найсильнішу агентну автономність для комплексних задач: самостійний пошук файлів, редагування, запуск тестів, підтримка subagents, checkpoints, hooks, skills, slash commands і MCP. У Task B він розв'язав однаковий bug-fix ticket за 49 секунд в 1 ітерацію, знайшов справжню причину бага, додав регресійні тести й не створив помітних регресій. У Task C Claude Code також єдиний із перевірених інструментів без додаткових підказок підхопив і root `AGENTS.md`, і `app/AGENTS.md`, що важливо для командних правил у monorepo або repo з nested проєктами.

Рекомендація: використовувати Claude Code як основний інструмент для ticket-to-code workflow, складніших багфіксів, рефакторингів, задач із запуском тестів і сценаріїв, де важлива автономність агента та дотримання project instructions.

## Рекомендований додатковий (secondary) інструмент

**Zed IDE** — для сценаріїв: швидкі точкові правки прямо в IDE, робота з кодом напряму. Він beginner-friendly, дає LSP підтримку без додаткових налаштувань, зрозумілий UX, знайомі shortcuts.

Чому: у Task A агент від Zed показав зручну IDE-інтеграцію, підтримку різних провайдерів моделей, MCP, project instructions/skills і зручний workflow всередині редактора. У Task B він також знайшов і виправив справжній баг з першого промпту за 55 секунд, але потребує уважнішого review, бо агент видалив один існуючий тест. Тому Zed варто використовувати як продуктивний IDE-native companion, але не як повністю автономний агент без перегляду diff.

OpenCode варто залишити як додаткову опцію для експериментів або budget/open-source сценаріїв: він був найшвидшим у Task B (25 секунд), легко стартував без додаткових налаштувань і коштував дешевше в recorded run, але потребував другої ітерації для перевірки фіксу та не підхопив nested `app/AGENTS.md` без явної підказки.

## Ключові компроміси (trade-offs)

- **Автономність vs контроль у редакторі:** Claude Code краще підходить для автономного multi-step execution, запуску тестів і ticket-to-PR стилю; Zed краще підходить для контрольованої роботи в IDE, коли розробник одразу бачить diff, buffers і LSP-контекст.
- **Якість результату vs швидкість:** OpenCode був найшвидшим на Task B, але потребував додаткового нагадування про перевірку. Claude Code був трохи повільнішим, зате дав найбільш якісне та лаконічне рішення з першої ітерації.
- **Ціна vs можливості:** OpenCode як open-source інструмент може бути дешевшим, якщо команда вже має власні API keys або internal AI gateway. Claude Code дає більше agentic-функцій і кращу інтеграцію з Claude-моделями, але може коштувати дорожче на великих контекстах або частому використанні.
- **Крос-tool портативність `AGENTS.md` vs tool-specific фічі:** Task C показав, що `AGENTS.md` не однаково автоматично підхоплюється всіма інструментами. Claude Code краще врахував nested instructions, тоді як Zed/OpenCode можуть потребувати явного посилання на `app/AGENTS.md` або додаткового налаштування.
- **Enterprise/privacy trade-off:** Claude Code має найсильніший enterprise/compliance профіль за матрицею, але залежить від Anthropic або обраного deployment path. Zed і OpenCode дають більше гнучкості у виборі провайдерів/локальних моделей, але enterprise-функції та data residency треба перевіряти окремо під конкретний deployment.

## Ризик/застереження, за якими варто стежити

**Ризик:** залежність від конкретного vendor/model ecosystem, особливо якщо команда зробить Claude Code основним інструментом і почне активно використовувати Claude-specific фічі: hooks, subagents, computer use, claude tag і т.д.

Що контролювати: тримати базові правила в portable `AGENTS.md`, документувати workflow так, щоб його можна було повторити в Zed/OpenCode, і раз на квартал переглядати pricing/limits та якість альтернативних інструментів. Для критичних змін обов'язково залишити human review diff + запуск тестів, незалежно від інструменту.
