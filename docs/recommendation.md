# Рекомендація для команди (Task D)

## Рекомендований основний (primary) інструмент

**Cursor**

Чому: На задачі BUG-101 (Task B) Cursor впорався швидше (1:30 проти 2:30) і з першого промпту знайшов баг, додав коректний фікс і регресійний тест — причому тест виявився суворішим (`expect(result.length).toBe(10)` замість `toBeLessThanOrEqual(10)`). За матрицею (Task A) Cursor сильніший у щоденній інтерактивній розробці: нативний IDE-досвід, inline diff, повна індексація кодової бази та multi-model provider (Claude, GPT, Gemini, DeepSeek). Для команди, де більшість роботи — правки в репозиторії з швидким review циклом, це оптимальний primary.

## Рекомендований додатковий (secondary) інструмент

**Claude Code** — для сценаріїв: headless-агенти в CI/CD, тривалі багатокрокові задачі (рефакторинг, міграції), CLI-автоматизація без IDE, subagents/Skills/Hooks для повторюваних workflow.

Чому: Task A показує, що Claude Code «agentic by default» — нативний agent harness з підтримкою CLI, CI/CD, Subagents і Skills, чого Cursor не має «з коробки». На BUG-101 якість фіксу була такою ж (1 промпт, баг знайдено), а Task C підтвердив, що кореневий `AGENTS.md` підхоплюється без додаткових налаштувань. Для задач поза IDE — nightly jobs, ticket-to-PR, складні pipeline — Claude Code доповнює Cursor, а не дублює його.

## Ключові компроміси (trade-offs)

- **Швидкість інтерактивної розробки (Cursor) vs агентна автономність поза IDE (Claude Code):** Cursor виграє на локальних правках і review; Claude Code — на headless-сценаріях і тривалих agent-сесіях.
- **Індексація кодової бази (Cursor) vs аналіз «на льоту» (Claude Code):** Cursor краще масштабується на великі монорепо завдяки семантичному пошуку; Claude Code покладається на контекстне вікно — на дуже великих проєктах це може коштувати більше токенів і часу.
- **Multi-model гнучкість (Cursor) vs глибина agent harness (Claude Code):** Cursor дозволяє перемикати провайдерів моделей; Claude Code прив'язаний до Anthropic, але має сильніший набір agent-примітивів (Hooks, Skills, Routines).
- **Крос-tool портативність (`AGENTS.md`) vs tool-specific фічі:** обидва інструменти читають кореневий `AGENTS.md` (Task C), але Cursor додатково підмішує user rules з глобальних налаштувань — команда має явно вирішити, що лежить у репо, а що в IDE-профілі, щоб уникнути розбіжностей між розробниками.

## Ризик/застереження, за яким варто стежити

**Vendor lock-in через розходження конвенцій між інструментами.** Якщо частина команди працює в Cursor (з user rules у профілі), а частина — у Claude Code (лише з `AGENTS.md` у репо), однаковий промпт може давати різну поведінку (наприклад, різна суворість тестів, як у BUG-101). Варто зафіксувати один canonical `AGENTS.md` у корені репозиторію, періодично верифікувати його в обох інструментах одним промптом (як у Task C) і не покладатися на tool-specific rules як єдине джерело правди.
