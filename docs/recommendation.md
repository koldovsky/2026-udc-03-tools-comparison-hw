# Рекомендація для команди (Task D)

> Синтез Task A + B + C.

## Рекомендований основний (primary) інструмент

**Codex**

Чому: за матрицею Task A Codex має сильний набір моделей GPT-5.2-GPT-5.5, підтримує `AGENTS.md`, skills marketplace, MCP-клієнт, інтеграції з GitHub/Slack і може працювати як MCP-сервер. За enterprise/privacy критеріями він також закриває базові потреби команди: SOC2, SSO/SCIM, data residency для enterprise-сценаріїв і режим, де Business/Enterprise/API дані не використовуються для навчання. Важливо, що слабке місце з Task A не є унікальним для Codex: як і Claude Code, він не має повної вбудованої семантичної індексації й покладається на файловий пошук, `AGENTS.md` та локальні евристики.

Практичні результати Task B і Task C роблять Codex кращим primary для щоденної розробки. У BUG-101 Codex був повільніший за Claude Code на маленькому багфіксі (приблизно 55 секунд проти 30 секунд), але якість результату суб'єктивно була кращою: інструмент з першого промпта знайшов справжню причину бага, додав regression coverage і ще врахував edge case `maxLength <= 0`. У Task C Codex краще підхопив саме локальний `app/AGENTS.md`: описав пакет текстових утиліт, команди `npm test` / `npm run typecheck`, відсутність lint і локальні кодові конвенції без додаткових нагадувань. Для щоденної командної роботи це важливіше за різницю в кілька десятків секунд.

## Рекомендований додатковий (secondary) інструмент

**Claude Code** — для сценаріїв: великі multi-file зміни, рефакторинги, задачі з довшим planning loop, headless/programmatic запуск, паралельна робота через worktrees або subagents.

Чому: у Task A Claude Code виглядає сильнішим саме в агентній автономності: Plan Mode, subagents, headless/programmatic режим і робота з Git worktrees краще підходять для довших задач, ніж для маленьких точкових правок. У Task B Claude Code був найшвидшим і також зробив робочий фікс з першого разу, тому його варто тримати як сильний secondary інструмент для складніших workflow.

## Ключові компроміси (trade-offs)

- **Якість фіксу vs швидкість:** Claude Code швидше закрив BUG-101, але Codex дав трохи якісніший результат і додатково покрив edge case.
- **Локальний контекст vs широкий repo context:** Codex краще дотримався nested `app/AGENTS.md`, а Claude Code при запуску з `app/` все одно орієнтувався на root `AGENTS.md`. Для монорепозиторіїв це може бути суттєво.
- **Агентна автономність vs контрольованість:** Claude Code має сильніші multi-step можливості, але може потребувати уважнішого контролю контексту. Codex у цьому завданні поводився більш передбачувано для локального пакета.
- **Індексація кодової бази:** за Task A в обох інструментів немає повної вбудованої семантичної індексації; обидва покладаються на файловий пошук, контекстні файли і локальні евристики.
- **Rules/skills/MCP:** Claude Code має глибші tool-specific можливості на кшталт `CLAUDE.md`, slash-команд, hooks і subagents; Codex краще вписується в простий крос-tool baseline через `AGENTS.md`, MCP і skills.
- **Ціна та ліміти:** обидва стартують приблизно з paid-tier рівня близько $20/міс, але мають різні ліміти, моделі й режими використання. Для команди це треба перевіряти під реальне навантаження, а не тільки на одній маленькій задачі.
- **Приватність/enterprise:** обидва мають enterprise-опції, SSO/SOC2 і режими, де дані клієнта не використовуються для навчання, але consumer/default налаштування відрізняються і потребують окремої перевірки перед роботою з чутливим кодом.

## Ризик/застереження, за яким варто стежити

Найконкретніший ризик з Task C — різна пріоритезація root і nested context files. Якщо інструмент бачить увесь репозиторій, він може виконувати root `AGENTS.md` замість локального `app/AGENTS.md`, що особливо небезпечно в монорепозиторіях або проєктах з різними правилами для пакетів. Мітігація: тримати root і nested `AGENTS.md` узгодженими, а в prompt для локальних задач явно вказувати, який context file є основним.
