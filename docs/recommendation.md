# Рекомендація для команди (Task D)

Синтез Task A (матриця можливостей) + Task B (та сама задача в кількох
інструментах) + Task C (крос-tool `AGENTS.md`). Усі твердження нижче спираються
на конкретні дані з цих трьох документів, а не на загальні враження.

## Рекомендований основний (primary) інструмент

**Claude Code (CLI від Anthropic)**

Чому:

- **Виграв Task B за швидкістю, вартістю й відсутністю тертя.** На тій самій
  задачі (`BUG-101`, `truncate()`) прогін на **Sonnet 5** дав робочий фікс
  найшвидше й найдешевше з усіх п'яти інструментів — **34 с API-часу, $0.24, 1
  промпт, без жодного permission-запиту** (`docs/comparison.md`). Знайшов
  справжню причину бага, а не симптом; регресій не було.
- **Найкраща якість кодування + явний контроль моделі й reasoning effort.**
  Родина Claude 5 дає осмислений вибір під задачу: Opus 4.8 (макс.
  reasoning) → Sonnet 5 (дефолтний баланс) → Haiku 4.5 (швидка/дешева). Це
  протилежність Copilot Free, звуженому до непрозорого `Auto` без вибору моделі
  (`docs/tool-matrix.md`).
- **Hooks — детермінований guardrail поза контекстом моделі.** Жорсткі правила
  на дії агента, які не витрачають токени й не залежать від того, чи модель
  «згадала» інструкцію — унікальна для Claude Code вісь надійності, якої немає в
  жодного іншого з перевірених інструментів (`docs/tool-matrix.md`, рядок
  «Унікальні переваги»).
- **Реальний enterprise-tier для командного впровадження.** SSO, **SOC 2 Type
  II**, керування командами; API/Enterprise-дані не використовуються для
  тренування. Це рівень гарантій, який можна принести в команду під compliance,
  а не лише особистий tier.

## Рекомендований додатковий (secondary) інструмент

**OpenCode (open-source CLI-агент)** — для сценаріїв:
консолідація навколо корпоративного проксі (один інтерфейс поверх OpenAI +
GitHub Copilot одночасно), CI / провайдер-агностичні пайплайни, ситуації, де
команда свідомо **не хоче прив'язки до однієї родини моделей**, потреба в
аудиті самого агента та в гранулярному контролі MCP-дозволів per-tool.

Чому:

- **Провайдер-агностичність — те, чого не дає primary.** 75+ провайдерів через
  один інтерфейс; на роботі учасника OpenCode одним CLI покриває і OpenAI, і
  GitHub Copilot через той самий внутрішній проксі, тоді як Claude Code
  розмовляє лише з Claude, а Codex CLI — лише з OpenAI (`docs/tool-matrix.md`,
  висновок). Один інтерфейс переживає зміну вендора.
- **Безпечний порядок edge-case перевірок + прозорість вартості.** У Task B
  виніс гілку `maxLength <= 0` окремо (як Antigravity й Codex) — на відміну від
  **обох** прогонів Claude Code, що цю гілку опустили; знайшов справжню причину
  з першого промпту без регресій. Має вбудовану context/cost-панель у UI
  (67 501 tokens, 17% контексту) — видно нативно, без сторонніх розширень.
- **Аудитованість і безкоштовність самого інструменту.** Повністю open source
  (можна форкнути й аудитувати), код/контекст не зберігаються на власних
  серверах OpenCode; оплата — лише за підключений провайдер. Плюс гранулярний
  permission-контроль аж до конкретного MCP-tool (`docs/tool-matrix.md`).

## Ключові компроміси (trade-offs)

- **Вибір моделі важить сильніше за вибір інструменту (ціна vs автономність).**
  У Task B той самий Claude Code на Opus 4.8 був найдорожчим і найповільнішим
  (**$0.89 / 2 хв 19 с**) проти Sonnet 5 (**$0.24 / 34 с**) — різниця більша,
  ніж між різними інструментами. Практично: дефолт команди — Sonnet 5, а Opus
  4.8 — свідомо, лише під складний reasoning, не «про всяк випадок».
- **Портативність `AGENTS.md` vs глибина tool-specific фіч.** Один узагальнений
  `AGENTS.md` підхопили всі 5 інструментів (Task C) — але Claude Code читає його
  **умовно**, а не безумовно, як `CLAUDE.md` (доведено canary-тестом). Крос-tool
  портативність коштує слабшої гарантії підхоплення; а найпотужніша фіча primary
  (hooks) — навпаки, tool-specific і непортовна на інші інструменти.
- **Якість/контроль моделі vs приватність за замовчуванням.** Claude Code дає
  SOC 2 Type II + вибір моделі, але дефолтний **Pro-акаунт може навчатися на
  даних** (треба вимикати вручну; API/Enterprise — ні). OpenCode власного SOC 2
  не має — успадковує enterprise-гарантії від підключеного провайдера, зате сам
  код не покидає його серверів і повністю auditable.
- **Вендор-специфічна глибина vs консолідація.** Claude Code прив'язаний до
  однієї родини моделей, але дає hooks/skills, яких немає в OpenCode; OpenCode
  жертвує цією глибиною заради одного інтерфейсу до багатьох провайдерів. Це і є
  вододіл між primary та secondary.
- **Прозорість discovery контексту.** Claude Code (явна цитата конвенції за
  номером) і Antigravity (візуальна панель rules/skills) показують, що підхопили
  контекст; Copilot Free — жодного індикатора, довелось підтверджувати окремим
  діагностичним промптом (Task C). Для командного onboarding видимість того, що
  саме читає агент, — недооцінена, але реальна перевага.

## Ризик/застереження, за яким варто стежити

**Vendor / model lock-in навколо primary + волатильність цінової політики.**
Claude Code розмовляє лише з родиною Claude, тож команда, яка стандартизується
на ньому, зав'язана на одного вендора: якщо Anthropic змінить ціни, ужорсточить
ліміти (Pro-план уже має **динамічний conversation budget з тимчасовим
кулдауном**) або зробить конкретну модель недоступною — fallback у межах того
самого інструменту відсутній.

Пом'якшення закладене прямо в цю рекомендацію: **secondary-вибір (OpenCode)
навмисно провайдер-агностичний** — один інтерфейс, який переживе зміну вендора,
тож команда не переписує весь workflow, а лише перемикає провайдера. Додатково
(Task C): не покладатися на portable `AGENTS.md` як на єдине джерело правил у
Claude Code через його **умовне** підхоплення — критичні конвенції та guardrails
дублювати в `CLAUDE.md` (безумовне system-level завантаження) або кодувати
hooks'ами.
