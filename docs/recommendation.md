# Рекомендації щодо впровадження AI-інструментів у команду (Task D)

**Мета:** Синтезувати результати порівняння можливостей інструментів (Task A), практичного усунення багів (Task B) та тестування крос-платформного контексту (Task C) у фінальне інженерне рішення для команди.

---

## 1. Первинний інструмент команди (Primary Tool) & Обґрунтування

**Вибір:** **Claude Code (CLI) / Agentic IDE (на базі моделей Anthropic)**

### Обґрунтування:
- **Найвища швидкість доставки коду (Time-to-Market):** Як показав практичний тест у Task B, інструмент здатний розв'язати баг, написати тести й провести верифікацію всього за **1 хвилину 5 секунд** у межах одного промпту.
- **Агентний цикл "під ключ" (Multi-step autonomy):** На відміну від класичних чатів, інструмент самостійно взаємодіє з файловою системою, запускає перевірки `npm run build` та `npm test`, аналізує їх результати й за потреби ітерує зміни до успішного проходження збірки та тестів. Це суттєво зменшує обсяг ручних дій розробника.
- **Глибока інтеграція контексту:** Інструмент нативно зчитує інструкції з `app/AGENTS.md` (Task C) на рівні файлової системи, що забезпечує 100% дотримання архітектурних конвенцій проєкту без додаткових нагадувань у чаті.

---

## 2. Вторинний інструмент (Secondary Tool) & Сценарії використання

**Вибір:** **ChatGPT Plus / Team (Web + Canvas)**

### Сценарії, де Secondary інструмент ефективніший за Primary:
- **Архітектурне проектування та брейнштормінг:** Коли задача ще не сформульована у вигляді чіткого квитка, а потребує обговорення архітектури, створення схем баз даних чи генерації концептів. Міркувальні моделі (серія `o1`/`o3-mini`) чудово працюють як "високорівневий спаринг-партнер".
- **Складне візуальне або контекстне редагування (Canvas):** Інтерфейс Canvas дозволяє зручно проводити рев'ю великих шматків коду, додавати коментарі та локально експериментувати з логікою без ризику зламати поточну робочу гілку в терміналі.
- **Швидкі ізольовані правки (Без IDE/CLI):** Коли розробнику потрібно швидко згенерувати регулярний вираз, написати ізольований скрипт міграції або розпарсити JSON-лог, не розгортаючи для цього локальне агентне середовище.

## 3. Ключові компроміси (Trade-offs)

### А. Ціна vs Автономність
Автономні агенти (Primary) використовують великі контекстні вікна та роблять багато послідовних запусків моделей (агентні петлі) під час дебагу. Це робить вартість однієї ітерації значно вищою (Usage-based оплата за токени через API). На противагу, ChatGPT (Secondary) пропонує фіксовану підписку (\$20/міс), але вимагає витрат дорогоцінного часу розробника на ручний копіпаст.

### Б. Портативність AGENTS.md vs Tool-specific фічі
Використання єдиного універсального `app/AGENTS.md` (Task C) дозволяє легко перемикатися між різними інструментами й не засмічувати репозиторій багатьма конфігами. Проте, ми втрачаємо можливість використовувати специфічні глибокі фічі конкретних екосистем (наприклад, кастомні prompt-шаблони, специфічні UI-кнопки чи внутрішні індекси плагінів).

### В. Якість індексації кодової бази vs Приватність
Глибока хмарна індексація репозиторію дає ШІ краще розуміння зв'язків у коді, але потребує оцінки політик обробки даних постачальника. Якщо використовуються хмарні сервіси, варто обирати тарифи з чітко задокументованими гарантіями, наприклад **OpenAI API** з політикою, за якою дані клієнтів не використовуються для навчання моделей за замовчуванням, або сервіси з підтримкою **Zero Data Retention**, якщо вони доступні для вашого сценарію. Локальний inference (наприклад, через Ollama) зменшує передачу даних стороннім сервісам, однак рівень приватності залежить від конфігурації середовища: налаштувань telemetry, логування, контролю доступу до хоста, а також довіри до моделей і компонентів, що використовуються (supply chain).

## 4. Критичний ризик / Застереження (Risk Assessment)

**Ключовий ризик:** **Контроль бюджету та неконтрольовані агентні петлі (Infinite Loops).**

Агентні інструменти мають право самостійно запускати команди в терміналі. Якщо агент потрапить у циклічну помилку (наприклад, фікс коду ламає тест $\rightarrow$ спроба виправити тест ламає лінтер $\rightarrow$ спроба виправити лінтер знову ламає код), він може за кілька хвилин витратити весь грошовий ліміт API-ключа, безкінечно перечитуючи великий контекст проєкту.

**Заходи безпеки:**
1. Обов'язково виставляти **жорсткі Hard Limits** (ліміти витрат на день/місяць) в панелі керування API-провайдера.
2. Обмежувати максимальну кількість кроків (max ітерацій) для автономного агента в межах однієї команди терміналу.
