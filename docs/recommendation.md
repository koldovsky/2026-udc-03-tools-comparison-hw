# Рекомендація для команди (Task D)

Синтез Task A (`tool-matrix.md`), Task B (`comparison.md`) та Task C
(`agents-md-notes.md`).

## Рекомендований основний (primary) інструмент

**Claude Code CLI**

На задачі Task B усі три інструменти знайшли справжній баг,
запропонували коректний фікс, додали регресійний тест і зробили це за 1
ітерацію — тобто відмінності немає. Вирішальними стали
дані з Task A: у Claude Code найзріліша екосистема контролю поведінки агента
(CLAUDE.md/memory, plan mode, Hooks, Skills, MCP, кастомні subagents) і
найдетальніша permission-модель — саме це підтвердилось у Task B, де Claude
Code явно питав дозвіл перед редагуванням файлів і запуском bash-команд, тобто
дає найкращу видимість/auditability дій агента "з коробки". Task C додатково
показав, що Claude Code дотримується того самого `AGENTS.md`, що й інші
інструменти, без якихось tool-specific доопрацювань — тобто вибір Claude Code
як primary не створює додаткового vendor-специфічного формату правил, який
довелось би підтримувати окремо.

## Рекомендований додатковий (secondary) інструмент

**Codex CLI** — для сценаріїв: headless-запуски в CI/CD, великі
багатофайлові рефактори з паралельними subagents, швидкі правки без відкритої
IDE.

Чому: у Task B Codex CLI виконав задачу повністю автономно, без жодних
проміжних дозапитів (на відміну від Claude Code, який зупинявся на
підтвердження edit/bash), — це саме та властивість, що потрібна для
безperешкодного headless-запуску в CI, де немає людини для відповіді на
permission-промпти. Вартість і якість результату в Task B були практично
ідентичні Claude Code ($0.168 проти $0.175), а з Task A відомо, що Codex
підтримує ту саму зв'язку AGENTS.md/rules/skills/MCP/subagents — тобто його
можна підключити як secondary без переписування конвенцій команди.

(Junie залишається кращим вибором лише для Java/Kotlin-проєктів у JetBrains
IDE завдяки PSI/semantic-індексації, але для поточного TypeScript/npm стеку
ця перевага не застосовна, а Task B показав операційне тертя — агент не зміг
одразу запустити тести й переплутав версію Node.)

## Ключові компроміси (trade-offs)

- **Ціна vs автономність.** Claude Code (\$20–\$200/міс або API pay-as-you-go)
  і Codex (\$20–\$100+/міс) дають повну агентну автономність (multi-step
  planning, subagents, запуск команд і тестів); Junie з кредитною моделлю
  вийшов дешевшим саме на цій задачі (~\$0.11 проти \$0.168–\$0.175), але це
  один data-point на тривіальній задачі, а не гарантія на масштабі.
- **Якість індексації vs приватність/enterprise-tier.** Junie використовує
  PSI/symbol navigation і опційний semantic indexing на embeddings — найкраща
  навігація кодовою базою, але передбачає передачу prompts/контексту зовнішнім
  LLM-провайдерам (пом'якшується BYOK/on-prem лише на AI Enterprise). Claude
  Code і Codex натомість покладаються на grep/файловий пошук — простіший і
  прозоріший потік даних, без окремого етапу індексації embeddings.
- **Портативність AGENTS.md vs глибина tool-specific фіч.** Task C
  підтвердив, що один `AGENTS.md` покриває всі три інструменти без
  доопрацювань — це знижує ризик прив'язки до одного вендора на рівні правил.
  Але ця портативність не поширюється на tool-specific можливості (Hooks і
  кастомні subagents у Claude Code, IDE-рефакторинги в Junie) — щоб ними
  скористатись, команді все одно доведеться свідомо "прив'язатись" до
  конкретного інструменту в конкретних сценаріях.
- **Permission-контроль vs швидкість.** Явні дозапити Claude Code перед
  edit/bash підвищують auditability, але додають тертя в інтерактивній
  роботі; повна автономність Codex прибирає тертя, але зменшує видимість
  проміжних кроків агента.

## Ризик/застереження, за яким варто стежити

**Непередбачуваність вартості при масштабуванні.** У Task B всі три
інструменти показали малий і схожий кошт (\$0.11–\$0.175) на одній маленькій
задачі — це не репрезентативно для довгих agentic-сесій з великими рефакторами
чи паралельними subagents, де ціноутворення (\$20–\$200/міс тарифи, API
pay-as-you-go, credit-системи на кшталт Junie) може розходитись у рази.
Потрібно прогнати командою більшу задачу (multi-file рефактор або тікет із
кількома ітераціями) перед тим, як фіксувати primary-вибір на рівні бюджету, а
не лише на рівні якості фіксу.
